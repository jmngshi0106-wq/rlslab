How is this different from the Supabase Security Advisor?
The Security Advisor tells you which tables are missing RLS. It cannot tell you why a policy you've already written is silently failing at runtime — returning empty results, causing HTTP 500s, or blocking Realtime subscriptions. Those are different problems. RLS Lab solves the second one.
Why not just ask ChatGPT to fix my RLS?
You've probably already tried this. The AI gave you SQL that looked correct — because it is syntactically correct. But it fails at runtime because AI doesn't know your calling context: whether you're using the anon client in a server component, whether your Prisma migration already wiped your grants, whether your Clerk users aren't in auth.users. AI without context generates the right-looking wrong answer. RLS Lab reads your actual schema and calling context — and finds the failure the AI can't see.
Will this work with Clerk + Supabase?
Yes. Clerk + Supabase is a specific failure class we handle explicitly. The core issue — auth.uid() returning NULL because Clerk users don't exist in auth.users — is one of the most common patterns we detect. We generate the correct current_setting('request.jwt.claims')::json->>'sub' replacement for every policy that needs it.
What if my schema is too complex for automated fixes?
The 48-hour manual guarantee covers this. If the automated fix doesn't resolve your specific failure, a human debugs it and delivers the corrected policy within 48 hours. If we can't fix it, your next month is free. Complex schemas are the exact use case this was built for — the patterns that break most often are the multi-tenant, org-scoped, role-based ones, not the simple ones.
